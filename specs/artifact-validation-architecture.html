Artifact Validation Engine

MVP Architecture Handoff — Serverless document validation via Bedrock Data Automation

product: eva

1. Project Overview

Asynchronously validates document artifacts (e.g., verifying if an uploaded image is a valid SSN card) submitted via Kuali forms.

Volume: 10s-100s forms/day Perf: Near real-time (async) Security: PII — purge immediately Infra: AWS Serverless + OpenTofu Lang: Python 3.13+ / ty strict / Powertools

2. Scope & Non-Goals

These are deliberate non-goals for the MVP, not oversights. Each is an accepted tradeoff to revisit as volume and requirements grow — don't build these unless a requirement forces it.

Non-goalWhy it's acceptable for MVP
No idempotency layerSQS is at-least-once, so an artifact may be validated and a result re-emitted more than once. Accepted — staging/purge is repeatable and downstream tolerates duplicate result events.
No reserved concurrencyRuns on default account concurrency. Volume is 10s–100s/day; revisit if throttling appears or volume grows.
Async only — no synchronous APIValidation is fire-and-forget via EEL. There is no request/response endpoint that returns a result inline.
15-min Lambda timeout is the hard ceilingA single artifact must validate within one invocation. No Step Functions or long-running orchestration in MVP.
Event Translator extraction is a separate workstreamEVA consumes from EEL. Do not build the translator inside this repo — it is its own product (see Components).
EVA is not a system of recordIt emits a result event and forgets. Persisting validation results is the downstream consumer's responsibility.
No DLQ auto-redriveFailed events sit in the DLQ for manual inspection/replay. No automated reprocessing in MVP.
Single region, no DRDeployed to us-west-2 only. No multi-region or disaster-recovery failover.

EVA Lambda Orchestrator Steps in Detail

  1. Validate Input: Pydantic model via Powertools event_parser. Reject malformed immediately.
  2. Retrieval: httpx to Kuali GraphQL API to retrieve form attachments (connect: 5s, read: 30s). Also extracts emplid and attempts to resolve document_type from the response metadata.
  3. Staging: S3 put. Key: {form_id}/{uuid}.
  4. Validation: BDA invocation with S3 URI.
  5. Immediate Purge: S3 delete the instant the result is received — the artifact is PII and must not linger. The 1-day S3 lifecycle rule is the backstop, not the primary mechanism.
  6. Emit Result: EEL event with form_id, document_type, source, isValidArtifact boolean. Scoped subscriptions via source. Always fires (pass and fail).
  7. IB Checklist Update (failure only): On validation failure, call Integration Broker to return the checklist item. get_checklists(emplid, "FPKGEX", "FMM40") → resolve seq3c → update_checklist_item(action="R"). Must-succeed — failure raises and retries via SQS.
  8. SFDC Create Closed Case (failure only): On validation failure, look up SFDC contact via get_contact_id(emplid), then create a closed CAR case. Triggers auto-response email to the student notifying them their document was returned. Must-succeed.
  9. Slack Notification (always): Post Block Kit message to webhook URL with validation outcome (pass/fail, form_id, document_type). Fire-and-forget — log warning on failure, do not block or retry. Consider dialing back to failures-only in future.

Conditional Execution Matrix

StepPassFailFailure Semantics
EEL emit✓✓Must-succeed (raises → SQS retry)
IB checklist update—✓Must-succeed (idempotent on retry)
SFDC closed case—✓Must-succeed (ordering prevents duplicates)
Slack notification✓✓Fire-and-forget (warning on failure)

Ordering rationale: EEL → IB → SFDC → Slack. IB is idempotent on retry. SFDC executes after IB succeeds, so on retry (IB success → SFDC failure → SQS redelivery) IB re-executes harmlessly and SFDC only fires once. Slack is last and non-blocking.

Document Type Resolution

The document_type field identifies what artifact was validated (e.g., SSN card, driver's license). Resolution strategy:

  1. Primary: Infer from the Kuali GraphQL API response during the retrieval step (field labels, attachment categories, or form metadata).
  2. Fallback: Static config mapping (form_id → document_type) if the GraphQL response does not carry usable metadata. Requires a deploy to add new form types.

5. Python Implementation Standards

HTTP Clients — httpx + stamina 
import stamina
import httpx

KUALI_TIMEOUT = httpx.Timeout(connect=5, read=30)

@stamina.retry(on=(httpx.TransportError, httpx.HTTPStatusError), attempts=3)
def fetch_attachment(url: str, api_key: str) -> bytes:
    response = httpx.get(
        url,
        timeout=KUALI_TIMEOUT,
        headers={"Authorization": f"Bearer {api_key}"},
    )
    response.raise_for_status()
    return response.content

    result = invoke_bedrock(s3_uri=staged_uri)

    logger.error("Bedrock throttled", extra={"form_id": form_id, "error": str(e)})

7. Observability

Standard All alerting via Datadog monitors → #prd-eva. CloudWatch Metrics are the data source; Datadog queries via AWS integration.

Why Datadog over CloudWatch alarms

  • Unified alerting — infra, APM, and log alerts route through one system with consistent escalation policies.
  • Richer conditions — anomaly, composite, and forecast monitors not available in CW alarms.
  • Better routing — PagerDuty/Slack integration, on-call schedules, and escalation chains in one place.
  • Correlation — alerts link directly to the relevant traces, logs, and dashboards in the same UI.

8. Security

Data Protection

  • S3: SSE-S3 encryption + aws:SecureTransport (HTTPS only) + block all public access
  • No PII in logs — form_id only
  • 1-day lifecycle rule as defense-in-depth

Secrets

  • Kuali API keys in Secrets Manager (rotation support)
  • Integration Broker password in Secrets Manager (Basic auth)
  • ServiceAuth client secret in Secrets Manager (for SFDC token acquisition)
  • Slack webhook URL in Secrets Manager
  • Validate at cold start — fail immediately if missing
  • Cache in memory for warm invocations

IAM

Principle of least privilege — scope to specific resource ARNs, actions, and conditions.

Key Scenarios

  • Valid event → validation passes → EEL emit + Slack only (no IB, no SFDC)
  • Valid event → validation fails → EEL emit → IB update → SFDC case → Slack
  • Malformed event → Pydantic rejects
  • Kuali timeout → stamina retries then raises
  • BDA throttle → boto3 adaptive handles
  • S3 delete fails → lifecycle guarantees purge
  • IB call fails → raises → SQS retry (SFDC/Slack never fire)
  • SFDC call fails → raises → SQS retry (IB is idempotent on re-execution)
  • Slack fails → warning logged, does not block or retry
