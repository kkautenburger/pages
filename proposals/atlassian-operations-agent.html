Proposal: Atlassian Operations Agent

An AI-powered assistant for the Atlassian team's daily operations

TL;DR: We propose building an AI agent that works alongside the Atlassian analysts — answering questions, executing routine tasks, auditing configurations, and automating repetitive workflows. The first capability shipped would be determined by the team based on what delivers the most value. The agent grows additional capabilities over time based on what you find most useful.

The Vision

The Atlassian team manages hundreds of projects, spaces, and user configurations across the enterprise. Much of this work is repetitive, well-defined, and follows documented standards — making it ideal for an agent-assisted workflow.

Rather than building a rigid CLI tool that does one thing, we're proposing a conversational agent that understands your domain and grows with your needs.

How It Works (Technically)

The agent is built on Kiro — the same AI assistant platform already used across Engineering. It consists of:

ComponentWhat It DoesAnalogy
Agent config Defines the agent's persona, scope, and guardrails Job description
Skills Domain knowledge — naming conventions, UPRD template, required fields, workflow steps Runbook / SOPs
Knowledge bases Indexed reference material the agent searches — product map, existing project inventory, team rosters, historical decisions Filing cabinet / institutional memory
Scripts Thin Python scripts that make API calls to Jira and Confluence The hands that do the work

The agent reads the skills to know what to do and how to validate, then calls scripts to execute. It always confirms with the analyst before making changes.

On Atlassian Rovo: Atlassian is developing Rovo — their own AI agent platform with native Jira/Confluence integration. As Rovo matures, we may replace this agent with Rovo, integrate it alongside Rovo, or continue using what we build here if the team finds it better meets their needs. This proposal is designed to deliver value now without waiting for Rovo's roadmap, and the skills and knowledge bases we create will be transferable regardless of the underlying platform.

Operations & Guardrails

QuestionHow It's Handled
How do we keep it from touching production by accident? Two layers. (1) The agent drafts every change and waits for explicit confirmation before any write — nothing is created, modified, or deleted without an analyst saying "go." (2) The agent inherits the permissions of whatever credential it runs under, so its blast radius is capped by that account's access. Destructive operations (delete project, remove space, bulk permission removal) can be disallowed outright in the agent's config so they're never even offered.
Does it log everything for audit? Yes. Every action runs through the Jira/Confluence REST API, which records the acting account in each item's history — same audit trail as a human using the UI. On top of that, the agent keeps a local session transcript (what was asked, what it proposed, what was confirmed, what API calls ran). Because writes are confirm-before-execute, there's always a human in the approval chain.
Who can use it? Access is gated entirely by credentials. The agent can only do what its connected account can do — so use is limited to whoever holds the service-account token (or runs it under their own Atlassian permissions). It is not a shared, always-on service; it runs for the person who launched it, with that person's (or the service account's) access.
Where does it run? Locally — on the analyst's own machine, the same way Engineering runs Kiro today. There's no new server to stand up, no dedicated infrastructure, and no agent sitting in the cloud with standing access to your tenant. Credentials stay on the local machine.
Who pays for it? The ESS team covers Kiro usage and the LLM token costs.

Templated Project Requests (22)All follow identical format: UPRD: {Name} (key: {code}). Submitted in batches of 2-4 on the same day.

How the Tempo Team Request flow actually works: When any project request is created — a Templated Project Request (UPRD/USVC) or a PMO Project (PRJ) — Jira Automation automatically spawns a separate "Tempo Team Request" Task, linked to the parent via a "split to" relationship. That automation only creates the ticket; an analyst must then manually go into Tempo, create the team, and assign it to the sponsoring Portfolio Manager.

Implication for automation: The agent creating a project request will not stop these tickets from being generated — the automation fires off the parent regardless of who created it. The opportunity is the reverse: have the agent fulfill the Tempo Team Request (create the team, assign the Portfolio Manager) instead of just generating busywork. This depends on whether Tempo exposes an API the agent can call — an open question for the team.

Patterns We Noticed

  • User Management dominates the queue — 42 tickets in 6 months (~7/month). Add/remove/deactivate users, grant access, vendor partners, ServiceNow RITM fulfillment. Clear inputs, consistent steps every time.
  • Templated Project Requests are highly structured — 22 tickets (~4/month), same format every time, always paired with a Tempo Team Request, submitted in batches
  • Tempo Team Requests are auto-generated follow-on tickets — 28 of the 49 Task tickets. Each one is spawned by Jira Automation when a project request (UPRD/USVC/PRJ) is created, then manually fulfilled by an analyst. Automating the project request doesn't remove them — but the agent could fulfill them (see flow above).
  • Ad-hoc ops work is varied but small — archive/restore projects, create boards, move issues, set up webhooks (~3-4/month)
  • Change Requests are infrequent but span a wide range — from 35-minute field updates to multi-week custom configuration work (8 in 6 months)
